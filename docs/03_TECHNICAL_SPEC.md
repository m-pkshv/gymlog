## 4. Архитектура
Слоистая, направление зависимостей только вниз:
```
UI (widgets, screens)            — только отображение и ввод; без бизнес-логики
  ↓
Controllers (Riverpod Notifier)  — состояние экрана, вызовы сервисов
  ↓
Services (application layer)     — бизнес-правила: статусы, валидация, прогрессия,
                                   рекорды, единицы, экспорт, активная тренировка
  ↓
Repositories (интерфейсы)        — контракт доступа к данным (D-13: изоляция хранилища)
  ↓
Data (drift database, DAO, seed) — реализация репозиториев
```
Правила:
- UI не обращается к репозиториям и БД напрямую.
- Все записи, затрагивающие несколько таблиц, — в одной транзакции drift.
- Domain-модели (immutable Dart-классы) отделены от drift-классов таблиц; маппинг в слое Data.
- DI — через Riverpod-провайдеры; единый composition root в `lib/app/providers.dart`.
- Ошибки: сервисы возвращают типизированные результаты (`Result<T, AppError>` — sealed-класс); исключения БД перехватываются в Data-слое и конвертируются в `AppError.storage`. UI-отображение ошибок — `04`, раздел 7.

### 4.1. Структура каталогов

## 5. Автосохранение
- Каждое подтверждённое изменение поля (уход фокуса, нажатие «готово», переключение чекбокса, завершение drag-and-drop) немедленно пишется через сервис в БД. Дебаунс текстовых полей — 500 мс, плюс форс-запись при уходе с экрана и при `AppLifecycleState.paused`.
- Отдельная кнопка «Сохранить» отсутствует везде, кроме модальных форм создания (упражнение, тег, измерение), где запись происходит по кнопке создания.
- Инвариант надёжности: убийство процесса в любой момент теряет не более последнего недописанного дебаунс-интервала одного текстового поля.

## 6. Единицы измерения (D-5)
- Хранение — всегда метрическое (`06`, раздел 3). Конвертация — единственный класс `core/units/unit_converter.dart`.
- Коэффициенты: 1 lb = 0.45359237 кг; 1 миля = 1609.344 м; 1 дюйм = 2.54 см.
- Отображение/округление: масса — 1 знак (кг) / 1 знак (lb); дистанция — км с 2 знаками или мили с 2 знаками (ввод в км/милях, хранение в м); замеры — 1 знак (см) / 1 знак (in); темп — `мм:сс` на км или на милю.
- Ввод: значение парсится в единицах текущей настройки и конвертируется в метрические до валидации диапазонов (диапазоны в `06` заданы в метрических).
- Переключение системы единиц меняет только отображение; повторная конвертация сохранённых значений не выполняется.
- Парсинг чисел в полях ввода принимает и точку, и запятую как десятичный разделитель независимо от локали.

## 7. Активная тренировка и таймеры

### 7.1. Модель времени
- Все расчёты — от «якорных» UTC-меток из `ActiveWorkoutState` (`06`, раздел 6.14), а не от тикающих в памяти счётчиков. Тик UI — раз в секунду, значение = f(now, якоря). Это делает восстановление после сворачивания/убийства процесса тривиальным.
- Длительность тренировки: `accumulatedActiveSec + (isPaused ? 0 : now − lastResumeAt)`. Пауза: перенос набежавшего в `accumulatedActiveSec`, установка `isPaused`.
- Таймер отдыха: хранится только `restTimerEndsAtUtc`. Остаток = `endsAt − now`; отрицательный — таймер истёк.

### 7.2. Сценарий режима
1. «Начать тренировку» → статус `inProgress`, создание `ActiveWorkoutState`, экран активной тренировки.
2. Отметка подхода выполненным → автосохранение; если `restTimerAutoStart` — запуск таймера отдыха на `defaultRestTimerSec` (изменяемо на месте: ±15 с, свой ввод).
3. Запуск таймера отдыха: планирование локального уведомления на `endsAt` («Отдых окончен — следующий подход»). Отмена/перезапуск таймера — отмена/перепланирование уведомления.
4. Сворачивание приложения: ничего не делаем — уведомление уже запланировано системе; таймеры восстановятся из якорей.
5. Возврат/перезапуск приложения: при старте, если есть `ActiveWorkoutState`, — баннер «Тренировка продолжается, N мин» с переходом в режим; состояние таймеров восстанавливается из якорей.
6. «Завершить» → подтверждение, если есть неотмеченные подходы («Отметить оставшиеся невыполненными и завершить?») → `completed`, `finishedAt = now`, `actualDurationSec` из формулы 7.1, удаление `ActiveWorkoutState`, отмена уведомлений → итоговый экран.
7. «Отменить» → подтверждение → `cancelled`, введённые данные сохраняются, `ActiveWorkoutState` удаляется.

### 7.3. Разрешения на уведомления (D-11)
- Запрос — контекстный: при первом запуске таймера отдыха (не при старте приложения) с предварительным пояснительным диалогом.
- Android 13+: `POST_NOTIFICATIONS` runtime; Android 12+: для точности использовать `zonedSchedule` c `AndroidScheduleMode.inexactAllowWhileIdle` (точное `exact` требует спец-разрешения — не используем; допустимая погрешность до ~1 мин; внутри открытого приложения таймер точен, т.к. считается от якорей).
- iOS: стандартный запрос alert+sound при первом использовании.
- Отказ: таймер работает внутри приложения; на экране таймера — ненавязчивая пометка «Уведомления выключены» со ссылкой в системные настройки. Повторный автоматический запрос не выполняется.

## 8. Статусы, копирование, шаблоны — бизнес-правила
- Переходы статусов — `06`, раздел 6.4.1; сервис `workout_service` — единственная точка смены статуса.
- «Скопировать предыдущую тренировку»: копируются упражнения, порядок, комментарии упражнений и **плановые** значения подходов; фактические значения, отметки выполнения, решения прогрессии — не копируются. Дата — выбранная пользователем, статус `draft`.
- «Скопировать показатели предыдущего выполнения» (в редакторе): фактические значения последнего завершённого выполнения этого упражнения подставляются в плановые значения текущих подходов (несовпадающее число подходов — добавляются/остаются как есть, лишние не удаляются).
- Шаблоны: создание из тренировки переносит структуру и плановые значения; создание тренировки из шаблона — обратное копирование (`06`, раздел 6.8).

## 9. Статистика: формулы и источники данных
Общие правила: источник — подходы с `isCompleted = 1` завершённых (`completed`, `isDeleted = 0`) тренировок (Этап 10, 2026-07-23: понятие «разминочный подход» удалено из приложения — раньше здесь было дополнительное условие `isWarmup = 0`, теперь все подходы равноценны); измерения — все неудалённые `BodyMeasurement`. Периоды графиков: 7 дн, 30 дн, 90 дн, 365 дн, всё время, пользовательский диапазон (включительно, по локальным датам). Пустой период — пустое состояние (`04`, раздел 6).

| Показатель | Формула / источник |
|---|---|
| Динамика веса тела | `BodyMeasurement` типа `body_weight`; точка = значение за день; линия по дням с записями (без интерполяции в БД, соединение точек на графике) |
| Динамика % жира | Аналогично, тип `body_fat` |
| Динамика замера | Аналогично, выбранный `MeasurementType` |
| Макс. вес (упражнение) | `max(actualWeightKg)` |
| Макс. повторения при весе W | `max(actualReps)` среди подходов с `actualWeightKg = W` (W выбирается из фактически использованных весов) |
| Тоннаж тренировки (strength/reps) | `Σ actualWeightKg × actualReps`; для `reps` без веса слагаемое 0 |
| Тоннаж за период | Сумма тоннажей тренировок периода |
| Расчётный 1ПМ | Epley (D-6): `w × (1 + r/30)`, r ∈ [1;12], w > 0; на графике — максимум за тренировку |
| Рекорды кардио | `maxDistance = max(actualDistanceM)`; `longestDuration = max(actualDurationSec)`; `bestPace = min(actualDurationSec / (actualDistanceM/1000))` при дистанции ≥ 500 м |
| Число тренировок | Count завершённых за период |
| Частота | Завершённые за период / число недель периода (1 знак) |

### 9.4. Прогрессия и счётчик стагнации (D-7)
Определения:
- «Результат тренировки по упражнению» = вектор показателей **лучшего рабочего подхода**: для `strength` лучший подход — с максимальным расчётным 1ПМ (при равенстве — больший вес); сравниваемые показатели: (вес, повторения). Для `reps`: (повторения, доп. вес). Для `cardio`: (дистанция, −время) — рост дистанции при не выросшем времени или снижение времени при не снизившейся дистанции. Для `time`/`stretch`: (длительность).
- «Рост»: существует показатель со строгим увеличением, и ни один показатель не уменьшился (сравнение с предыдущей завершённой тренировкой, содержащей это упражнение). NULL-показатель считается «не уменьшился», если был NULL и остался NULL; переход значение→NULL считается уменьшением.
Алгоритм пересчёта (полный, идемпотентный):
1. Взять все завершённые тренировки с упражнением, отсортировать по (`date`, `finishedAt`).
2. `count = 0`; для каждой пары (prev, cur): если «рост» — `count = 0`, иначе `count += 1`.
3. Записать `count` в `ExerciseProgressionState`. Первая тренировка счётчик не увеличивает.
Сброс/инвалидация: любые изменения истории по упражнению (см. триггеры `06`, раздел 6.10) → полный пересчёт по упражнению. Пропущенные/отменённые тренировки не участвуют.
`progressionDecision` — ручная отметка пользователя, на счётчик не влияет; UI показывает подсказку «N тренировок без роста» из счётчика.

## 10. Экспорт CSV (D-9)

### 10.1. Общее
- Результат — один ZIP `gymlog_export_YYYY-MM-DD_HHmm.zip`, передаётся в `share_plus` (открыть в Google Sheets, сохранить в Drive/Files — средствами системы).
- `formatVersion = 1`. Кодировка UTF-8 **с BOM** (для корректного открытия в Excel/Sheets). Разделитель — запятая; экранирование RFC 4180. Десятичный разделитель — точка; даты `YYYY-MM-DD`; моменты — UTC ISO 8601. Независимость от RU/EN локали гарантируется собственными форматтерами (никаких `NumberFormat` текущей локали при записи).
- Пустое значение — пустая ячейка (не `null`, не `0`).
- Экспортируются только неудалённые данные; архивные упражнения — экспортируются (история на них ссылается).
- Операция журналируется в `ImportExportOperation`; сборка — во временный каталог, ZIP формируется полностью и только затем передаётся в шаринг (частичный файл наружу не попадает).

### 10.2. `manifest.json`
`formatVersion` v2 (Этап 10, 2026-07-23): убрана колонка `is_warmup` из `workouts.csv` (10.3) — понятие разминки удалено из приложения. v3 (Этап 10, 2026-07-23): убрана колонка `comment` из `measurements.csv` (10.4) — комментарий к замеру удалён из приложения.
```json
{
  "formatVersion": 3,
  "appVersion": "1.0.0",
  "exportedAtUtc": "2026-07-19T14:00:00Z",
  "files": {
    "workouts": "workouts.csv",
    "measurements": "measurements.csv",
    "exercises": "exercises.csv"
  },
  "counts": {"workouts": 0, "sets": 0, "measurements": 0, "exercises": 0}
}
```

### 10.3. `workouts.csv` — строка = подход (порядок колонок фиксирован)
`exercise_name, exercise_id, exercise_type, workout_id, workout_date, workout_name, workout_status, workout_tags, workout_comment, workout_duration_sec, exercise_order, exercise_comment, progression_decision, set_number, is_completed, planned_weight_kg, planned_reps, actual_weight_kg, actual_reps, rpe, rir, planned_duration_sec, actual_duration_sec, planned_distance_m, actual_distance_m, resistance, incline_percent, avg_heart_rate, side, set_comment`
- `exercise_name` — первая колонка (требование D-9); для встроенных — каноническое английское имя (`06`, раздел 12).
- `workout_tags` — имена тегов через `;` внутри одной ячейки.
- Булевы — `true`/`false`. Тренировка без упражнений — одна строка с пустыми полями подхода. Сортировка строк: `workout_date`, `workout_id`, `exercise_order`, `set_number`.

### 10.4. `measurements.csv`
`date, type, custom_type_name, value, unit, source` — `type` — слаг встроенного либо `custom`; `unit` ∈ `kg|percent|cm`; значения всегда метрические.

### 10.5. `exercises.csv`
`exercise_id, name, type, is_built_in, is_archived, primary_muscle, secondary_muscles, equipment, description, youtube_url` — `secondary_muscles` через `;`.

### 10.6. Задел под импорт (пост-MVP)
Формат разработан симметричным для импорта: стабильные `exercise_id`/`workout_id` позволят детектировать дубликаты; `formatVersion` — эволюцию формата. Требования импорта из ТЗ (предпросмотр, валидация, отчёт об ошибках, режимы дубликатов, транзакционность, бэкап перед импортом) фиксируются как обязательные для этапа импорта; отдельная спецификация — до начала того этапа.

## 11. Надёжность и безопасность (сквозные требования)
1. Потеря данных недопустима: автосохранение (раздел 5), транзакции, якорные таймеры (раздел 7), Undo/подтверждения (`06`, раздел 10).
2. Офлайн: ни одна основная функция не обращается к сети. Сеть в MVP используется только неявно (открытие YouTube-ссылки во внешнем браузере).
3. Миграции — `06`, раздел 11; тест миграций обязателен в каждом этапе, меняющем схему.
4. Основные данные — только SQLite (D-18). БД — в каталоге документов приложения; резервное копирование ОС (Android Auto Backup / iCloud) — включено для файла БД (проверить размер < лимитов Auto Backup; при превышении — исключить и полагаться на экспорт; решение зафиксировать на Этапе 10).
5. Секретов в клиенте нет; появление любого ключа (будущие интеграции) — только через отдельное решение владельца.
6. Производительность: списки — `ListView.builder`/`SliverList`; графики — предагрегация в сервисе, не в виджете. Целевые метрики: холодный старт < 2 с на среднем устройстве, кадр < 16 мс на скролле истории из 1000 тренировок (профилировать на Этапе 10). Требование «запросы истории — с пагинацией (страница 50 тренировок)» снято на Этапе 10 (2026-07-22, решение владельца): замер `WorkoutRepository.watchHistory` на 1000 тренировках/9000 подходах в файловой БД дал 71 мс на первую эмиссию — `ListView.builder`, уже используемый на экране «История», сам по себе даёт виртуализацию скролла; полноценная пагинация (LIMIT/OFFSET + infinite-scroll UI) не даёт измеримой пользы на целевом масштабе и не реализовывалась.
7. Логирование: локальный логгер (уровни debug/info/error), без отправки наружу; в release — только error в файл с ротацией 1 МБ.
