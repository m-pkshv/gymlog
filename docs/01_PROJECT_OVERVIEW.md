# 01_PROJECT_OVERVIEW — Обзор проекта «GymLog» (дневник тренировок)

## 1. Назначение документа
Единая точка входа в проектную документацию. Определяет цели продукта, границы MVP, зафиксированные решения, глоссарий и карту документов. Все остальные документы ссылаются на решения из раздела 6 и термины из раздела 8; не дублируют их.

## 2. Область ответственности
- Продуктовые цели и границы (что входит / не входит в MVP).
- Журнал принятых решений (Decision Log) — единственный источник истины по решениям.
- Глоссарий — единственный источник истины по терминам.
- Открытые вопросы верхнего уровня.

Не входит: техническая архитектура (см. `03_TECHNICAL_SPEC.md`), модель данных (см. `06_DATA_MODEL.md`), UI (см. `04_UI_UX_SPEC.md`), план работ (см. `02_DEVELOPMENT_PLAN.md`), правила работы ИИ-исполнителя (см. `05_AI_INSTRUCTIONS.md`).

## 3. Продукт
Мобильное приложение — дневник тренировок в спортзале. Работает полностью офлайн; все пользовательские данные хранятся локально на устройстве. Платформы: Android (API 26+) и iOS (15+). Только портретная ориентация.

### 3.1. Главные пользовательские сценарии
1. Спланировать будущую тренировку (с нуля, из шаблона, копией прошлой).
2. Провести тренировку в специальном режиме: отмечать подходы, вести таймер отдыха, записывать фактические результаты.
3. Просматривать историю тренировок с фильтрами.
4. Отслеживать прогресс по упражнениям и личные рекорды.
5. Вести дневник веса, процента жира и замеров тела.
6. Экспортировать все данные в CSV через системное меню «Поделиться».

### 3.2. В объёме MVP
- Каталог упражнений: 20 встроенных + пользовательские; поиск, фильтры, карточка, архивирование.
- 5 предопределённых типов упражнений (силовое, кардио, на повторения, на время, растяжка). Произвольный конструктор типов — **вне MVP** (см. D-14).
- Тренировки со статусами, тегами, комментариями; подходы с плановыми/фактическими значениями.
- Режим активной тренировки: таймеры, восстановление состояния, локальные уведомления таймера отдыха (D-11).
- Шаблоны тренировок.
- Измерения тела (ручной ввод).
- Статистика и графики (формулы — `03_TECHNICAL_SPEC.md`, раздел 9).
- Личные рекорды (вычисляются из истории, D-8).
- Экспорт CSV (формат — `03_TECHNICAL_SPEC.md`, раздел 10).
- Локализация RU/EN, светлая/тёмная темы, кг/фунты (D-5).

### 3.3. Вне MVP (последующие этапы)
- Импорт CSV (предпросмотр, валидация, дубликаты, транзакционность, бэкап перед импортом).
- Прямая интеграция с Google Sheets — только после отдельного подтверждения пользователя.
- Импорт из Apple Health / Google Health Connect (отдельный этап; перед реализацией — отдельный документ про разрешения, отказ пользователя, различия платформ).
- Облачная синхронизация (архитектура закладывает готовность: D-13).
- Изображения для пользовательских упражнений; анимации/GIF (D-3).
- Планшетные макеты (адаптивная вёрстка телефонов покрывает планшеты).

## 4. Целевая аудитория и контекст использования
Пользователь находится в спортзале: одна рука занята, мокрые/уставшие руки, частые прерывания (звонки, сворачивание приложения), возможное отсутствие сети. Следствия для UX: крупные зоны нажатий, минимум обязательного ввода, автосохранение после каждого изменения, восстановление состояния активной тренировки, все действия доступны без жестов (жесты — только ускорители).

## 5. Роли ИИ-исполнителя
ИИ, реализующий проект, совмещает роли: системный аналитик, архитектор, senior mobile-разработчик (Flutter), UI/UX-дизайнер, специалист по БД, QA-инженер, технический писатель. Правила переключения ролей и рабочий процесс — `05_AI_INSTRUCTIONS.md`.

## 6. Журнал принятых решений (Decision Log)
Формат: ID — решение — обоснование. Изменение любого решения требует явного подтверждения владельца проекта и записи здесь.

| ID | Решение | Обоснование / примечания |
|---|---|---|
| D-1 | Стек: **Flutter (канал stable) + Dart**. Точные версии Flutter/Dart и всех пакетов фиксируются в начале Этапа 0 в `pubspec.yaml` / `pubspec.lock` и не обновляются без отдельной задачи. | Кроссплатформенность, нативное поведение клавиатуры/списков, богатая экосистема. Заменяет исходно рассматривавшийся Unity по решению владельца от 2026-07-19. |
| D-2 | Хранилище: **SQLite через пакет `drift`** (реляционная схема, транзакции, типобезопасные запросы, встроенный механизм миграций `schemaVersion`). | Надёжность, целостность связей, миграции. |
| D-3 | Изображения: встроенные упражнения — статичные PNG/WebP в ассетах; пользовательские упражнения в MVP **без изображений** (поле в схеме зарезервировано). Анимации — вне MVP. | Минимизация объёма работ и размера сборки. |
| D-4 | Встроенная база: **20 упражнений**. Контент (названия, описания, ссылки, изображения) предоставляет владелец проекта по шаблону JSON-сида (`06_DATA_MODEL.md`, раздел 12) в ходе Этапа 2. До получения контента используется заглушечный сид из 5 упражнений. | Контент — работа владельца; разработка не блокируется. |
| D-5 | Единицы: одна настройка «метрическая / имперская». Внутреннее хранение всегда метрическое: масса — кг, дистанция — м, замеры — см, длительность — секунды. Конвертация только при вводе/отображении. Коэффициенты и округление — `03_TECHNICAL_SPEC.md`, раздел 8. | Единообразие, отсутствие двойных конвертаций. |
| D-6 | Расчётный 1ПМ — формула **Epley**: `1RM = вес × (1 + повторения / 30)`. Разминочные подходы исключаются; подходы с повторениями > 12 не участвуют в оценке 1ПМ. Значение всегда помечается как «расчётное». | Простая, общепринятая формула. |
| D-7 | Прогрессия: «результат вырос», если хотя бы один показатель лучшего рабочего подхода увеличился и **ни один** не уменьшился относительно предыдущей завершённой тренировки с этим упражнением. Иначе счётчик стагнации +1; при росте — сброс в 0. Точные правила по типам упражнений — `03_TECHNICAL_SPEC.md`, раздел 9.4. | Критерий владельца проекта. |
| D-8 | Личные рекорды **вычисляются из истории**; в БД хранится только кэш-таблица `PersonalRecord` с пересчётом при изменении затрагивающих данных. | Редактирование/удаление истории не порождает «висячих» рекордов. |
| D-9 | Экспорт CSV: строка = подход; первая колонка — название упражнения. Три CSV-файла (тренировки, измерения, каталог упражнений) + `manifest.json` с версией формата, упакованные в ZIP. UTF-8 с BOM, даты ISO 8601, десятичный разделитель — точка независимо от локали. Полная спецификация — `03_TECHNICAL_SPEC.md`, раздел 10. | Формат владельца + переносимость и основа будущего импорта. |
| D-10 | Минимальные ОС: **Android 8.0 (API 26)**, **iOS 15**. | Покрытие устройств + современные API уведомлений. |
| D-11 | Локальные уведомления таймера отдыха — **в MVP** (пакет `flutter_local_notifications`; runtime-разрешение на Android 13+, запрос разрешения iOS при первом использовании таймера). | Таймер отдыха без уведомлений бесполезен при свёрнутом приложении. |
| D-12 | Язык документации — русский; идентификаторы, имена сущностей, полей, кода, файлов — английские. | Читаемость + переносимость кода. |
| D-13 | Готовность к синхронизации без её реализации: UUID v4 для всех сущностей, поля `createdAt`/`updatedAt` (UTC), мягкое удаление `isDeleted`, слой репозиториев, изолирующий хранилище от бизнес-логики. | Требование ТЗ о будущей облачной синхронизации. |
| D-14 | В MVP нет произвольного конструктора типов упражнений; 5 фиксированных типов с фиксированными наборами метрик. Поле `statsMetrics` (метрики, участвующие в статистике) в MVP определяется типом и не редактируется пользователем; в схеме зарезервировано. | Требование ТЗ: не усложнять архитектуру, статистику и импорт. |
| D-15 | Статусы «черновик» и «запланирована» различаются явным действием пользователя «Запланировать»; активной (`inProgress`) может быть только одна тренировка одновременно. | Однозначность жизненного цикла. |
| D-16 | Шаблоны хранятся в отдельных таблицах (не как тренировки с флагом), содержат только плановые значения, не участвуют в истории и статистике. | Изоляция от статистики по построению, а не по фильтрам. |
| D-17 | State management — **Riverpod**; навигация — **go_router**; графики — **fl_chart**; локализация — ARB + `flutter_localizations`/`intl`. Полный список пакетов — `03_TECHNICAL_SPEC.md`, раздел 3. | Зрелые, документированные пакеты; предположение, помечено как решение по умолчанию — владелец может заменить до конца Этапа 0. |
| D-18 | Основные данные — только в SQLite. `SharedPreferences`/`NSUserDefaults` допустимы исключительно для нечувствительных UI-мелочей (например, последняя открытая вкладка). Секреты и ключи в клиенте не хранятся (в MVP секретов нет). | Аналог запрета PlayerPrefs из ТЗ. |
| D-19 | Удаление пользователем: тренировки, шаблоны, измерения, теги — мягкое удаление с возможностью «Отменить» (snackbar, 5 с) и/или подтверждением. Пользовательские упражнения, использованные в истории, — только архивирование. Встроенные упражнения — только архивирование, физическое удаление невозможно. Полные правила — `06_DATA_MODEL.md`, раздел 10. | Требования надёжности ТЗ. |

## 8. Глоссарий
| Термин | Значение |
|---|---|
| Упражнение (Exercise) | Единица каталога: жим лёжа, бег и т. п. Встроенное или пользовательское. |
| Тип упражнения (ExerciseType) | Один из 5 фиксированных типов, определяющий набор метрик подхода. |
| Тренировка (Workout) | Событие с датой, статусом, списком упражнений и подходов. |
| Упражнение тренировки (WorkoutExercise) | Вхождение упражнения в конкретную тренировку (с порядком, комментарием, решением о прогрессии). |
| Подход (ExerciseSet) | Одна запись выполнения: план/факт по метрикам типа упражнения. |
| Рабочий подход | Подход с `isWarmup = false`. Только рабочие подходы участвуют в статистике, рекордах и прогрессии. |
| Шаблон (WorkoutTemplate) | Заготовка тренировки с плановыми значениями; вне истории и статистики. |
| Тег (WorkoutTag) | Метка тренировки; можно скрыть в настройках без удаления данных. |
| Измерение тела (BodyMeasurement) | Запись веса, процента жира или замера с датой, значением, источником, комментарием. |
| Личный рекорд (PersonalRecord) | Лучший показатель по упражнению, вычисляемый из истории (кэшируется). |
| Решение о прогрессии | Отметка на упражнении тренировки: увеличить / повторить / снизить / не выбрано. |
| Счётчик стагнации | Число последовательных завершённых тренировок с упражнением без роста результата (D-7). |
| Активная тренировка | Тренировка в статусе `inProgress`; работает режим выполнения с таймерами. |
| Вертикальный срез | Минимальный сквозной сценарий Этапа 1 (`02_DEVELOPMENT_PLAN.md`). |
| Сид (seed) | Начальные данные, загружаемые в БД при первом запуске (встроенные упражнения, типы замеров). |
