# 02_DEVELOPMENT_PLAN — План разработки

## 1. Назначение документа
Разбивает проект на этапы, каждый из которых завершается работоспособной сборкой на Android и iOS. Для каждого этапа: цель, функциональность, задачи, зависимости, затрагиваемые системы, результат, критерии приёмки, ручные проверки, автотесты, риски, условия перехода. Технические детали — по ссылкам на `03_TECHNICAL_SPEC.md` (TS), `04_UI_UX_SPEC.md` (UX), `06_DATA_MODEL.md` (DM).

## 2. Область ответственности
Последовательность и объём работ, контроль качества этапов. Не входит: содержание технических решений (TS/DM/UX).

## 3. Сквозные правила всех этапов
1. Этап закрыт только при выполнении **всех** критериев приёмки и зелёных автотестах; порядок работы ИИ — `05_AI_INSTRUCTIONS.md`.
2. Каждый этап заканчивается сборкой: `flutter build apk --debug` и `flutter build ios --no-codesign` без ошибок; приложение запускается и проходит ручной smoke-чек-лист этапа.
3. Любое изменение схемы БД: +1 к `schemaVersion`, запись в DM 11.1, миграционный тест «БД прежней версии с данными → миграция → данные целы».
4. Регресс: ручные проверки предыдущих этапов из «критического набора» (помечены ★) повторяются на каждом следующем этапе.
5. Новые экраны сразу создаются с состояниями загрузки/пусто/ошибка (UX 6), строками в обоих ARB и поддержкой обеих тем — «доделаем потом» запрещено.
6. Unit-тесты обязательны для: сервисов статусов, прогрессии, рекордов, конвертера единиц, статистики, генератора CSV, миграций.

## 4. Этапы

### Этап 0 — Каркас проекта
**Цель:** компилируемый скелет с БД, навигацией, темами, локализацией.
**Функциональность:** пустые экраны 5 вкладок с навигацией; переключение темы/языка вручную в коде.
**Задачи:** инициализация проекта (портрет-only, applicationId рабочий `dev.gymlog.app` до Q-2); фиксация версий пакетов (TS 3); `analysis_options.yaml`; структура каталогов (TS 4.1); drift-схема версии 1 полностью по DM (все таблицы — сразу, чтобы не плодить миграции); сиды справочников и заглушечный сид 5 упражнений (DM 12); слои Result/AppError, UnitConverter (TS 6); go_router с маршрутами (UX 4); темы из seed-цвета (UX 9); ARB-каркас RU/EN; логгер (TS 11.7).
**Зависимости:** нет. **Системы:** все базовые.
**Результат:** приложение запускается, вкладки переключаются, БД создаётся и сидится.
**Критерии приёмки:** сборки обеих платформ ОК; при первом запуске в БД присутствуют справочники и 5 упражнений; повторный запуск не дублирует сид; `flutter analyze` — 0 ошибок.
**Ручные проверки:** запуск на Android-эмуляторе и iOS-симуляторе; смена системной темы меняет тему приложения. ★
**Автотесты:** unit UnitConverter (кг↔lb, м↔мили, см↔in, округления); тест создания БД и сидов; тест идемпотентности сида.
**Риски:** несовместимость версий пакетов → фиксировать `pubspec.lock` сразу.
**Переход:** все критерии выполнены.

### Этап 1 — Вертикальный срез ★
**Цель:** сквозной минимальный сценарий ТЗ, доказывающий жизнеспособность архитектуры.
**Функциональность (ровно по ТЗ):** создание пользовательского упражнения (только имя + тип); создание тренировки; добавление упражнения в тренировку; добавление подходов (план/факт, разминка, ✓); автосохранение; закрытие и повторное открытие с сохранёнными данными; завершение тренировки; отображение в списке истории.
**Задачи:** репозитории и сервисы Exercise/Workout (минимум); упрощённые S-02 (список без фильтров), S-03 (без тегов/статус-меню/прогрессии — только draft→completed), S-06 (список + FAB), S-08 (имя+тип); автосохранение с дебаунсом (TS 5); переходы `draft → inProgress → completed` через кнопку «Завершить» (без экрана S-04 — он в Этапе 4; «Начать» пока сразу помечает inProgress в редакторе).
**Зависимости:** Этап 0. **Системы:** data, services, features/history, workout_editor, exercises.
**Результат:** пользователь проходит сценарий целиком без потери данных.
**Критерии приёмки:** сценарий среза проходит на обеих платформах; убийство процесса посреди ввода теряет не более одного дебаунс-интервала (TS 5); история показывает завершённую тренировку с корректной датой и числом упражнений; невозможно две тренировки `inProgress` (DM 6.4.1).
**Ручные проверки ★:** пройти срез; убить приложение после ввода веса → перезапустить → вес на месте; создать упражнение из редактора тренировки → оно сразу добавлено.
**Автотесты:** unit-сервис статусов (разрешённые/запрещённые переходы); integration-тест среза (создать→добавить→подходы→завершить→прочитать из БД); widget-тест таблицы подходов (ввод, ✓ копирует план в факт — DM 6.7).
**Риски:** архитектурные ошибки всплывут здесь — это дёшево; НЕ добавлять функциональность сверх среза.
**Переход:** срез стабилен, критерии выполнены. **Только после этого** расширение функциональности (требование ТЗ).

### Этап 2 — Каталог упражнений полностью
**Цель:** полный каталог по ТЗ.
**Функциональность:** поиск, фильтры (тип/мышцы/оборудование/архив/пользовательские), карточка S-07 (с историей выполнений), полная форма S-08 (все поля DM 6.1), архивирование, правила удаления (DM 10), YouTube-ссылка, изображения встроенных, «прошлые результаты» и «копировать показатели прошлого выполнения» в редакторе (TS 8).
**Задачи:** полный сид-пайплайн из `exercises_v1.json` + `ExerciseL10n`; интеграция контента владельца (Q-1) или продолжение на заглушках; блокировка смены типа при истории (DM 6.1).
**Зависимости:** Этап 1. **Системы:** exercises, workout_editor, data/seed.
**Критерии приёмки:** поиск находит по RU и EN имени встроенного; фильтры комбинируются; архивное упражнение скрыто из выбора, но открывается из старой тренировки; встроенное нельзя удалить (нет такого действия), пользовательское с историей — только архив; неиспользованное удаляется физически с подтверждением.
**Ручные проверки ★:** архивировать использованное упражнение → старая тренировка отображается корректно.
**Автотесты:** unit правил удаления/архивирования; тест обновления сида (новое упражнение добавляется, существующее обновляется); widget-тест поиска.
**Риски:** контент владельца задерживается → этап закрывается на заглушках, интеграция контента — отдельной задачей позже (не блокирует).
**Переход:** критерии выполнены.

### Этап 3 — Тренировки полностью
**Цель:** весь жизненный цикл тренировок по ТЗ.
**Функциональность:** статусы и все переходы (DM 6.4.1); теги (создание, назначение, скрытие через настройку `showTags`); копирование тренировки; создание «с нуля/из шаблона (заглушка до Этапа 5)/копией»; перенос даты; фильтры истории (диапазон, название, статусы, несколько тегов); календарь-вид; reorder упражнений drag + меню; комментарии; Undo-удаления; решение о прогрессии (ручная отметка) и счётчик стагнации (TS 9.4) с подсказкой.
**Зависимости:** Этап 2. **Системы:** history, workout_editor, services (progression), settings (только флаг showTags).
**Критерии приёмки:** запрещённые переходы статусов недоступны в UI и отвергаются сервисом; фильтр по 2+ тегам работает (режим ИЛИ; предположение — зафиксировано); скрытие тегов убирает чипы/фильтр, повторное включение возвращает всё; копия тренировки не содержит фактов (TS 8); счётчик стагнации соответствует правилам D-7 на подготовленных данных.
**Ручные проверки ★:** сценарий «скопировать прошлую → перенести дату → провести»; reorder кнопками без жеста.
**Автотесты:** unit прогрессии (таблица кейсов: рост/не рост/NULL-переходы по каждому типу упражнения); unit фильтров истории (запросы DAO); widget-тест Undo.
**Риски:** производительность истории → пагинация с Этапа 1, профилировка при 1000 записей (скрипт-генератор тестовых данных сделать здесь).
**Переход:** критерии выполнены.

### Этап 4 — Активная тренировка
**Цель:** режим выполнения с таймерами и восстановлением (TS 7).
**Функциональность:** S-04 полностью; таймер тренировки с паузой; таймер отдыха (±15 с, пропуск, автозапуск по настройке); локальные уведомления + контекстный запрос разрешений (TS 7.3); восстановление после сворачивания и убийства процесса; завершение с обработкой неотмеченных подходов; S-05 итог; запрет второй активной.
**Зависимости:** Этап 3. **Системы:** active_workout, notification_service, ActiveWorkoutState.
**Критерии приёмки:** свернуть на 10 мин → таймеры показывают верное время (якоря, не тики); убить процесс при активной → запуск предлагает продолжить, длительность корректна; уведомление приходит при свёрнутом приложении по окончании отдыха (допуск TS 7.3); отказ в разрешении не ломает таймер внутри приложения; пауза исключается из `actualDurationSec`.
**Ручные проверки ★:** полная тренировка «в поле»: старт → отметки → отдых с уведомлением → пауза → завершение → итог → история. На обеих платформах. Проверка Android 13+ (runtime-разрешение) и Android 8 (без него).
**Автотесты:** unit расчёта времени по якорям (включая паузы и восстановление); unit «завершить с неотмеченными»; integration восстановления состояния.
**Риски:** различия поведения уведомлений по вендорам Android → допуск неточности зафиксирован (TS-2); iOS-симулятор не показывает часть уведомлений → проверять на устройстве при возможности, иначе зафиксировать как известное ограничение проверки.
**Переход:** критерии выполнены.

### Этап 5 — Шаблоны
**Цель:** шаблоны по ТЗ (DM 6.8, TS 8).
**Функциональность:** S-12/S-13; создание с нуля и из тренировки; тренировка из шаблона (с выбором даты); дублирование; архивирование; изоляция от истории/статистики.
**Зависимости:** Этап 3 (редактор переиспользуется). **Критерии приёмки:** шаблон не появляется в истории и не влияет на статистику/рекорды (проверка запросов); создание из тренировки переносит структуру и планы, но не факты; цикл «тренировка → шаблон → новая тренировка» сохраняет порядок и значения.
**Ручные проверки:** цикл выше ★.
**Автотесты:** unit копирования в обе стороны; тест изоляции (запросы статистики не видят шаблоны — по построению, но тест фиксирует).
**Риски:** дублирование кода редактора → выделить общие виджеты таблицы подходов.
**Переход:** критерии выполнены.

### Этап 6 — Измерения тела
**Цель:** дневник измерений (DM 6.9, 5.3).
**Функциональность:** S-14/S-15; ввод веса, % жира, стандартных и пользовательских замеров; правило «одна запись на тип в день» с заменой; управление пользовательскими типами (создание/архив/удаление по DM 10); отображение в единицах настройки.
**Зависимости:** Этап 0 (+навигация «Ещё»). **Критерии приёмки:** ввод 75.5 кг при имперской настройке (ввод в lb) сохраняет метрику корректно (сквозная проверка D-5); дубль за день предлагает замену; пользовательский тип с записями не удаляется — только архив.
**Ручные проверки ★:** переключение единиц меняет отображение всех измерений без изменения хранимых значений (проверить экспортом позже).
**Автотесты:** unit валидации диапазонов по unitKind; unit правила дубликата дня.
**Риски:** низкие. **Переход:** критерии выполнены.

### Этап 7 — Статистика и рекорды
**Цель:** все показатели ТЗ по формулам TS 9.
**Функциональность:** S-09/S-10; графики динамик (вес, % жира, замеры); прогресс по упражнению (макс. вес, 1ПМ Epley, тоннаж, повторения при весе); рекорды кардио; число/частота тренировок; периоды (6 шт., включая пользовательский); кэш `PersonalRecord` с триггерами пересчёта (DM 6.10); показ новых рекордов на итоговом экране S-05.
**Зависимости:** Этапы 4, 6. **Критерии приёмки:** значения на графиках совпадают с ручным расчётом на подготовленном наборе данных (эталонный датасет в тестах); редактирование старой тренировки пересчитывает рекорды; разминочные подходы и незавершённые тренировки не влияют; периоды и пользовательский диапазон фильтруют корректно; пустые периоды — пустые состояния.
**Ручные проверки ★:** изменить вес в старом подходе → рекорд обновился; графики читаемы в тёмной теме.
**Автотесты:** unit каждой формулы TS 9 на эталонном датасете (главный тестовый актив этапа); unit триггеров пересчёта кэша.
**Риски:** производительность агрегаций → SQL-агрегация в DAO, не в Dart-циклах по всем строкам; профилировать на 1000 тренировок.
**Переход:** критерии выполнены.

### Этап 8 — Экспорт CSV
**Цель:** экспорт по спецификации TS 10.
**Функциональность:** S-16; генерация трёх CSV + manifest → ZIP → share; журнал операций; справка о формате.
**Зависимости:** Этапы 3, 6 (данные). **Критерии приёмки:** ZIP открывается; `workouts.csv` импортируется в Google Sheets без искажений (кириллица, даты, числа) при RU и EN локали устройства; строки соответствуют TS 10.3 (порядок колонок, сортировка, теги через `;`); значения метрические независимо от настройки единиц; тренировка без упражнений даёт одну строку; операция журналируется, сбой (эмуляция нехватки места) даёт `failed` без частичного файла наружу.
**Ручные проверки ★:** экспорт на устройстве с русской локалью → открыть в Google Sheets → выборочно сверить 10 значений с приложением.
**Автотесты:** golden-тест генератора CSV (эталонный датасет → байт-в-байт сравнение с эталонными файлами, включая BOM); unit экранирования RFC 4180 (запятые, кавычки, переводы строк в комментариях).
**Риски:** различия share-таргетов по устройствам → системная ответственность, вне контроля приложения.
**Переход:** критерии выполнены.

### Этап 9 — Настройки, локализация, темы, доступность (полировка)
**Цель:** довести сквозные качества до требований UX.
**Функциональность:** S-17 полностью; смена языка/темы/единиц на лету; проверка всех экранов на RU-длины строк, обе темы, масштаб текста 1.6×, TalkBack/VoiceOver (UX 11–12); пустые/ошибочные состояния — ревизия; палитра тегов; подтверждение UX-1/UX-2 у владельца.
**Критерии приёмки:** чек-лист UX 9–12 пройден по каждому экрану (таблица результатов в PR); нет захардкоженных строк (`flutter analyze` + поиск литералов) и цветов вне темы.
**Ручные проверки ★:** прогон основных сценариев на RU+тёмная+масштаб 1.3.
**Автотесты:** тест полноты ARB (все ключи есть в обоих файлах); widget-тесты критичных экранов в обеих темах (smoke).
**Переход:** критерии выполнены.

### Этап 10 — Стабилизация и релизная подготовка
**Цель:** релиз-кандидат.
**Задачи:** полный регресс ★-чек-листов всех этапов; профилирование (TS 11.6: старт < 2 с, скролл 1000 тренировок без jank); решение TS-1 (бэкап БД) и Q-2 (имя, иконка, id); сплеш/иконки; release-сборки (`appbundle`, ipa) с обфускацией Dart (`--obfuscate --split-debug-info`); прогон миграционного теста от версии 1 до текущей; финальная вычитка документации (техписатель) и генерация README со сборочными инструкциями.
**Критерии приёмки:** release-сборки собираются и проходят smoke на устройствах; все открытые вопросы документов закрыты или явно перенесены с решением владельца; краш-фри прогон полного регресса.
**Переход:** релиз.

## 5. Пост-MVP (последовательность зафиксирована, детализация — перед стартом)
