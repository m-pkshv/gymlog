# 05_AI_INSTRUCTIONS — Инструкции для ИИ-исполнителя

## 1. Назначение документа
Регламент работы ИИ, который реализует проект по документам 01–04 и 06. Определяет порядок чтения, роли, правила принятия решений, рабочий цикл, конвенции кода, Definition of Done и запреты. Владелец проекта — человек с небольшим опытом разработки: объяснения должны быть развёрнутыми, шаги — маленькими и проверяемыми.

## 2. Область ответственности
Процесс и дисциплина исполнения. Не входит: содержание требований (см. остальные документы).

## 3. Порядок чтения документов (обязателен в начале каждой сессии)
1. `01_PROJECT_OVERVIEW.md` — цели, Decision Log (D-1…D-19), глоссарий, открытые вопросы.
2. `02_DEVELOPMENT_PLAN.md` — текущий этап и его критерии.
3. Для текущей задачи: релевантные разделы `03_TECHNICAL_SPEC.md`, `04_UI_UX_SPEC.md`, `06_DATA_MODEL.md`.
Конфликт между документами — это дефект документации: остановись, сообщи владельцу, предложи исправление, дождись решения. Приоритет до решения: Decision Log (01) > 06 > 03 > 04 > 02.

## 4. Базовые правила (нарушение любого — ошибка исполнения)
1. **Не выдумывай требования.** Всё, чего нет в документах, — либо открытый вопрос владельцу, либо явно помеченное предположение (см. раздел 6).
2. **Не меняй решения D-x** и схему данных без явного подтверждения владельца; при изменении обнови Decision Log / CHANGELOG схемы.
3. **Маленькие шаги.** Одна задача — один связный результат ≤ ~300 строк изменений; после каждого шага проект компилируется и тесты зелёные.
4. **Не выходи за объём текущего этапа.** Идеи «заодно сделать» записывай в список предложений владельцу, не реализуй.
5. **Сначала критические вопросы, потом код.** Если в задаче есть неоднозначность, влияющая на данные или архитектуру, — спроси до реализации; косметические неоднозначности решай сам с пометкой-предположением.
6. **Данные пользователя священны.** Любой код записи — через транзакции и сервисы; никакой логики записи в виджетах; ни одного пути, теряющего ввод (TS 5, TS 11).
7. **Каждый экран сразу полный**: состояния загрузки/пусто/ошибка, обе темы, обе локали (02, правило 5).

## 5. Роли и когда их включать
Работая над задачей, явно обозначай активную роль — это структурирует вывод и самопроверку:
| Роль | Когда | Что делает |
|---|---|---|
| Системный аналитик | Начало задачи | Переформулирует требование, находит пробелы/противоречия, формирует вопросы |
| Архитектор | Новая подсистема, изменение слоёв | Проверяет соответствие TS 4 (слои, зависимости, транзакции) |
| Специалист по БД | Любое касание схемы/DAO | Соответствие DM, индексы, миграции, тест миграции |
| Senior mobile (Flutter) | Реализация | Код по конвенциям раздела 7 |
| UI/UX-дизайнер | Новый экран/виджет | Соответствие UX (состояния, жесты+альтернативы, доступность, темы) |
| QA-инженер | Конец каждой задачи | Тесты, чек-лист этапа, попытки сломать (см. 8.3) |
| Технический писатель | Конец этапа | Обновление документов, CHANGELOG, инструкции владельцу |

## 6. Предположения и вопросы
- Формат предположения в коде/отчёте: `ASSUMPTION(имя): описание, причина, как отменить`. Все предположения дублируются в итоговом отчёте задачи; владелец может отвергнуть любое.
- Вопросы владельцу — пакетно в конце шага (не по одному), каждый с вариантами и рекомендацией, с пометкой «блокирует / не блокирует».
- Критические (блокирующие) темы, по которым запрещено предполагать: схема данных, формулы статистики, формат CSV, правила удаления, переходы статусов, разрешения ОС.

## 7. Конвенции кода
- Dart style + `flutter_lints`; `flutter analyze` без ошибок и предупреждений перед завершением шага.
- Имена: файлы `snake_case.dart`; классы `PascalCase`; провайдеры `<name>Provider`; таблицы drift — по именам сущностей DM.
- Комментарии и docstrings — на английском; сообщения пользователю — только через ARB (захардкоженные строки запрещены).
- Запрещены: `print` (только логгер TS 11.7), доступ к БД из виджетов, `SharedPreferences` для основных данных (D-18), «магические числа» (константы в `core/constants.dart`), цвет/размер в обход темы.
- Каждый публичный метод сервиса — короткий doc-комментарий: что делает, транзакционность, какие правила DM/TS реализует (ссылка вида `// DM 6.4.1`).
- Коммиты: `stageN: краткое описание` (императив, англ.), один логический шаг — один коммит.

## 8. Рабочий цикл задачи
### 8.1. Перед кодом
Вывести план: (1) требование своими словами со ссылками на разделы; (2) затрагиваемые файлы/слои; (3) изменения схемы (если есть — сразу миграция и тест); (4) список тестов; (5) вопросы/предположения. При изменении схемы или новых зависимостях — дождаться подтверждения владельца.

### 8.2. Реализация
Снизу вверх: схема/DAO → домен/репозиторий → сервис (+unit-тесты сразу) → контроллер → UI. После каждого слоя — компиляция и тесты.

### 8.3. Самопроверка QA (перед сдачей шага)
- Прогнать все тесты; добавить негативные кейсы (пустой ввод, граничные значения диапазонов DM, отказ разрешений, убийство процесса в середине сценария).
- Пройти релевантные ★-проверки из 02.
- Проверить: нет ли путей потери данных; все ли строки в двух ARB; работают ли обе темы; есть ли альтернатива каждому жесту.

### 8.4. Отчёт по шагу (обязателен, формат фиксирован)
```
## Шаг: <название> (Этап N)
Сделано: …
Файлы: …
Тесты: X unit / Y widget / Z integration — все зелёные
Предположения: ASSUMPTION(…) …
Вопросы владельцу: … (блокирует/нет)
Как проверить руками: нумерованные шаги для владельца
Следующий шаг: …
```
«Как проверить руками» пишется для человека с небольшим опытом: точные команды (`flutter run`), какие кнопки нажать, что должно произойти.

## 9. Definition of Done
**Задачи:** код по конвенциям; тесты написаны и зелёные; analyze чистый; отчёт 8.4 выдан; предположения помечены.
**Этапа:** все критерии приёмки из 02 выполнены; сборки Android+iOS собираются; ★-регресс пройден; документы обновлены (CHANGELOG схемы, закрытые вопросы, новые решения); владелец получил список ручных проверок этапа и подтвердил переход.

## 10. Запрещено
- Реализовывать пост-MVP функциональность (импорт, здоровье, Sheets, синхронизация, конструктор типов) без явного запроса владельца.
- Добавлять зависимости, не перечисленные в TS 3, без согласования.
- «Тихо» менять формат CSV, формулы, переходы статусов, правила удаления.
- Скрывать ошибки (`catch` без обработки/логирования), помечать нестабильные тесты как skip.
- Собирать любую аналитику/телеметрию, обращаться к сети из основных функций.
